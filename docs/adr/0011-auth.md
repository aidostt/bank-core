# ADR-0011: JWT RS256 + JWKS, refresh sessions with rotation, RBAC

Status: accepted · Date: 2026-07-18

## Context
The gateway must validate tokens without calling identity per request; refresh
tokens must be revocable; roles are few.

## Decision
identity signs access tokens (RS256, TTL 15m, claims: sub, roles, sid) and exposes
`/.well-known/jwks.json`. Gateway validates locally with a cached JWKS (refetch on
unknown kid). Refresh tokens: 256-bit random, stored hashed (SHA-256) in `sessions`,
rotated on every use; presenting a rotated (already-used) token revokes the whole
session family (theft detection). Roles: customer, support, admin — checked at the
gateway per route and re-asserted in services for ownership.

## Alternatives
HS256 shared secret — one secret in every service, no key rotation story.
Full OAuth2/OIDC server (ory/keycloak) — realistic for enterprises, but hides the
auth engineering this project wants to show; roadmap.

## Consequences
Key pair generated at first boot into a mounted volume; kid embedded; rotation
procedure documented in identity service doc.
